## Problem Statement

当前仓库的 chat / SOP 生成链路，在知识库检索不到可用 citation 时会直接进入 `no_context`，在生成模型重试失败时才进入 `retrieval_fallback`。这对“运营/流程负责人”这类内部用户来说，问题不是答案不够聪明，而是流程在最需要继续推进的那一刻被中断。

本次要解决的不是“把系统变成一个随时联网的通用 Agent”，而是给现有 RAG 系统增加一个受控的外部知识补答出口，让用户在知识库没有覆盖时仍能继续完成当前任务，同时不破坏现有知识库证据边界、权限边界和主契约稳定性。

## Demand Evidence

- 用户明确指出，当前痛点来自“知识库没有的东西答不上来”，希望系统能接入外网 API 做补答。
- 目标用户被收敛为内部“运营/流程负责人”，不是泛化的全部问答用户。
- 当前最真实的损失不是直接业务事故，而是效率损失，用户在 `no_context` 后需要自己重新找资料，流程被打断。

这说明需求不是“更强的模型能力”，而是“在空上下文时不要让会话断掉”。

## Status Quo

- 现状：知识库命不中时，系统直接返回 `no_context`。
- 用户行为：当前更像“流程卡住”，而不是系统自动切到另一个可靠信息源继续回答。
- 竞争对象：不是另一个更强的问答系统，而是“用户放弃当前入口，自己去别处查”。

## Target User & Narrowest Wedge

目标用户：
- 内部运营/流程负责人
- 他们关心的是流程能不能继续推进，而不是系统能不能无限制联网

最小切口：
- 第一版只做 `chat-only`
- 只在 `no_context` 或明确失败分支触发外网补答
- 知识库命中时，仍然完全保持现有 RAG 主路径
- 外网结果与库内 citation 严格分离，不伪装成知识库证据
- `SOP` 生成保留为二期，不与第一版同发

## Constraints

- 不应把 `backend/app/services/retrieval_service.py` 变成会主动联网的黑盒，避免污染当前 retrieval 排序主线。
- 第一版优先复用现有 `chat_service.py`、`LLMGenerationClient`、`system_config_service.py`、telemetry / request snapshot / runtime gate。
- 必须保留现有 `no_context`、`retrieval_fallback` 语义的可观测性，新增独立 `external_fallback` 模式。
- 外部结果必须保留来源 URL、来源类型、抓取时间、触发原因，不能混进现有知识库 citation 语义。
- `external_sources` 必须作为正式 contract 字段进入 chat 响应，而不是偷偷塞进 metadata。
- 外网补答必须允许灰度关闭，并且使用独立配置面，不复用现有 `external_api_*` 模型路由配置。
- 第一版只允许受控 provider 和受控来源策略，不做开放浏览器或无限制 web 搜索。
- 第一版必须定义显式失败语义，至少区分 `no_result`、`provider_error`、`policy_blocked`。
- 第一版必须有硬延迟预算，并支持短 TTL cache，避免空上下文时把等待时间和外部成本一起放大。
- 需要考虑项目已知坑：当 `RAG_LLM_SOURCE=api` 时，`system_config.json` 中残留的本地模型路由可能导致外部模型配置漂移，这说明本方案不仅是代码改动，也是配置一致性问题。

## Premises

1. 外网补答应该只挂在 `no_context` / 明确失败分支，不直接改核心 retrieval 排序。  
2. 外网结果必须和知识库 citation 分开标记，不能伪装成库内证据。  
3. LangChain 可以使用，但更适合作为工具封装层，而不是接管现有 service 控制面。  

以上前提已在本次 office-hours 中确认。

## Approaches Considered

### Approach A: 服务层受控补答

在 `chat_service.py` 与 `sop_generation_service.py` 的 `no_context` / 明确失败路径中调用新的 `ExternalKnowledgeService`。该服务负责调用外网搜索 API、网页摘要 API 或可信外部知识 API，再由现有生成链路生成答案或摘要。LangChain 可用于封装某个搜索 provider 或规范工具调用，但整体触发、回退、日志与响应模式仍由现有服务层掌控。

优点：
- 最贴合当前代码结构，复用度最高
- 不需要重织 retrieval 主线
- 易于灰度、回退、审计

缺点：
- 第一版只覆盖明确空上下文场景
- 后续若要多步推理或多工具编排，可能需要再上移一层抽象

### Approach B: LangChain / LangGraph 主编排

把知识库检索、低置信度判定、外网工具、答案生成都放进统一 agent graph，让模型决定何时调用工具、如何组合证据。

优点：
- 扩展性最强，后续接更多工具方便
- 业界示例多，LangGraph 官方也有 CRAG 风格教程

缺点：
- 会引入第二套控制面
- 现有 trace、telemetry、contract、权限边界都要重新梳理
- 延迟、误触发、越权和调试成本显著上升

### Approach C: 仅做外部检索建议

在 `no_context` 时不直接调用外网回答，而是给用户一个受控的外部查询建议或一键跳转。

优点：
- 最安全
- 几乎不碰主契约和主流程

缺点：
- 用户仍需自己完成最后一步
- 不能真正解决“流程断掉”的核心痛点

## Recommended Approach

推荐 Approach A。

原因：
- 它最准确地命中了当前问题本质，即在 `no_context` 时提供可控补答，而不是把整个系统升级成 Agent 平台。
- 它最大程度复用了现有代码资产，尤其是 `chat_service.py` / `sop_generation_service.py` 中已经存在的失败分支语义。
- 它把 LangChain 放在合适的位置上，作为工具封装层，而不是新的系统中枢。

建议的第一版形态：
- 第一版只覆盖 `chat`
- 新增 `ExternalKnowledgeService`
- 架构只保留两层：`ExternalKnowledgeService` + `provider adapter`
- LangChain 若使用，只允许藏在 provider adapter 内部，不泄漏到 chat service 对外接口
- 新增独立配置开关与配置面，例如 `external_fallback_enabled`
- 仅在 `no_context` 或显式失败时触发
- 新增独立响应模式 `external_fallback`
- 新增正式外部来源结构 `external_sources`
- 明确 UI 文案，告诉用户“以下内容来自外部来源，不是上传知识库证据”
- 增加短 TTL cache，只缓存 `external_fallback` 的规范化结果

建议数据流：

```text
user question
    |
    v
ChatService
    |
    +--> CitationPipeline -> citations found ------> existing RAG path
    |
    +--> no citations / explicit failure
             |
             v
      ExternalKnowledgeService
             |
             +--> provider adapter (optional LangChain wrapper)
             |
             +--> source policy / allowlist / sanitization
             |
             +--> normalized external_sources
             |
             v
      ChatResponse(mode=external_fallback, external_sources=[...])
```

## Open Questions

- 第一版具体接哪一个 provider，搜索 API、可信知识 API，还是两者之一？
- 外网结果是只做摘要，还是允许直接基于外部结果生成自然语言答案？
- 权限边界如何体现，例如某些部门是否禁止外网补答？
- `SOP` 二期复用 chat 的 provider / source policy，还是需要独立策略？

## Success Criteria

- 当知识库命中时，用户体验与现有 RAG 路径完全一致
- 当知识库未命中时，系统能返回一个明确标记为外部来源的补答结果，而不是直接 `no_context`
- 外部来源和知识库 citation 在响应结构和前端展示上严格区分
- `external_sources` 被正式纳入 chat 契约，并同步更新前后端类型和主契约测试
- 外部补答失败时，至少能区分 `no_result`、`provider_error`、`policy_blocked`
- 外网补答可通过配置一键关闭
- 所有触发都能在 telemetry / snapshot / trace 中被观测到
- 外部补答总预算控制在 2 到 4 秒内，超时明确回退
- 重复 no-context 查询不会无节制击穿外部 provider
- 用户在空上下文问题上的流程中断率显著下降

## Distribution Plan

本方案面向现有 Web 服务，不新增新的独立交付物。分发路径沿用当前 API 与前端发布链路。

上线方式建议：
- 先通过配置开关在内部环境灰度
- 先接单一可信 provider
- 先只开放给 chat 场景，再决定是否同步给 SOP 生成场景
- 先不开开放浏览器，不做 agent 自主联网

- 契约测试：
  - `ChatResponse` / `ChatStreamMetaPayload` 新增 `external_sources`
  - 前端 `types.ts` 与 client 同步
  - `MAIN_CONTRACT_MATRIX.md` 与主契约测试同步

- 来源策略测试：
  - fake provider fixture 覆盖缺失 URL、域名不允许、空内容、脏标题、异常 payload

## The Assignment

先不要写 LangChain 代码。先做一件事：把你想接的“外网 API”限定成一个非常具体的来源集合，最多 1 到 2 个，并明确回答这三个问题：

1. 它回答的是什么类型的问题？  
2. 它返回的原始证据是什么样子，URL / 标题 / 摘要 / 结构化字段？  
3. 为什么你愿意让它在这个系统里出现，而不是别的来源？  

如果这三个问题答不清，系统最后会变成“为了联网而联网”。

## What I noticed about how you think

- 你一开始不是问“怎么做智能 Agent”，而是问“知识库没有的东西怎么办”，这说明你盯的是断点，不是炫技。
- 你接受了“只在 `no_context` 时触发”的最小切口，说明你更在意先把真实问题补上，而不是一口气把系统做成全功能平台。
- 你点名了 LangChain，但并没有执着于“必须全栈 LangChain 化”，这说明你对工具是务实态度，不是技术信仰。
