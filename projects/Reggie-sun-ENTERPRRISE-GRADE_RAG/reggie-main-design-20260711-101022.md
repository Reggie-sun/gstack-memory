## Problem Statement

`Enterprise-grade_RAG` 已具备大量 retrieval、chat、citation、memory 与权限能力，但当前 chat architecture convergence 仍在推进，部分运行时证明尚未闭环。用户希望在完成当前收敛任务后尽快让生产技术部门试点，同时利用纯 AI 开发的高效率继续快速迭代。

真正的问题不是开发速度不够，而是验证面过多且彼此分散。一个改动可能通过 pytest、route diagnostics 或 citation count，却仍在完整可见正文、多轮 document scope、sync/stream parity、permission 或 runtime identity 上失败。当前 Phase 2 worktree 的 named-overview switch 也暴露了同一问题：所谓“关闭”只跳过 specialized route，然后落入普通生成，并没有真正关闭不可靠能力。

本设计要建立一个极薄的生产技术部门试点合同：保留工艺工程师真正需要的复杂多轮语义；不可靠能力必须用户可见地 fail closed；新增一个 validator-only 的快速 gate，把现有验证真源组合成单一入口，但不创建第二套 route、truth 或 release control plane。

## Demand Evidence

- 已有具体工艺工程师拉动内部试点，不是纯粹的技术展示。
- 当前典型替代流程是放弃系统回答后回到 PDF、共享盘或原文手工核对，一次通常多花 10–20 分钟。
- 用户明确要求短期上线，并愿意暂时关闭部分能力来换取可信的试点边界。
- 当前仍以内部 AI/QA 测试为主，尚未完成一次工艺工程师不受引导的完整多轮观察，因此需求证据存在，但真实使用证据仍不足。
- 试点的首要价值是查找效率，不是直接替代安全审批或质量责任人。

## Status Quo

### User Workflow

当回答看起来不可信时，工艺工程师会重新打开原文，搜索关键词、定位章节并人工核对。系统没有稳定节省这一步时，真实竞争对手不是另一个 AI 产品，而是“自己翻原文”。

### Engineering Workflow

仓库已有 `verify-retrieval-focused`、`verify-retrieval-preflight`、`verify-permission-visibility-gate`、`eval_chat_memory.py --behavioral` 和 `verify-retrieval-release` 等验证入口，但开发窗口仍需要临时拼接 sample IDs、runtime identity、四账号权限与可见正文审阅。高开发速度因此转化为高验证漂移风险。

## Target User & Narrowest Wedge

### Target User

首个试点角色是生产技术部门的工艺工程师。

他的核心任务不是让系统替他做审批，而是在已授权的企业文档中更快完成参数、步骤、检查点和文档范围核对。

### Narrowest Wedge

首发不是单轮问答，也不是全功能 Chat。它保留以下复杂多轮语义：

1. 询问一份明确文档中的具体参数、步骤或要求。
2. 后续省略文档名时，仍保持同一 document scope。
3. 用户显式切换到新文档时，清除旧 scope。
4. 标题歧义时先返回正式 clarification，用户选择后下一轮才锁定 scope。
5. 每个 enabled answer turn 的最终正文都使用目标 citation 中的具体业务事实，不能只返回泛化定位或模板套话。
6. Portal 的 sync/stream 最终 `answer`、`mode`、`citations`、`external_sources` 与 `clarification_request` 语义一致。

### Capabilities Temporarily Off

- named-document whole-document overview：用户可见地关闭，不允许回落普通 LLM/RAG 继续猜。
- chapter LLM composer / section summarization：保持关闭，provider capacity 与 artifact readiness 只作为重新启用条件。
- Agent Shadow visible mutation：保持 auditor-only，不改 answer、mode、citations 或 clarification。
- external search：不属于本轮试点合同；pilot 账号的 UI 不暴露开关，API 请求强制 `enable_external_search=false`，不得通过内部 fallback 偷跑外网。

### Pilot Capability Exposure Matrix

所有 pilot 可达能力必须落入下表之一，不允许出现“用户可访问，但没有任何 gate”的第四种状态。

| Capability | Pilot exposure | Required proof | Active Owner |
| --- | --- | --- | --- |
| explicit document fact/detail | enabled + fast-gated | target document、具体正文事实、citation alignment | Answer Dispatch + Evidence Enforcement |
| same-document follow-up | enabled + fast-gated | inherited scope、正文事实、target citation | Intent Graph / RetrievalPlan scope |
| explicit new-document reset | enabled + fast-gated | stale scope cleared、new target locked | Intent Graph / RetrievalPlan scope |
| ambiguous title clarification and selection | enabled + fast-gated | safe options、selection 后才锁定 scope | Answer Dispatch + formal clarification renderer |
| Portal sync/stream final semantics | enabled + fast-gated | final answer/mode/citation/clarification parity | final response owner；stream 是 transport |
| named-document whole-document overview | disabled + fast-gated fail-closed | `capability_disabled`、no generation、no source leakage | Answer Dispatch |
| chapter LLM composer / section summarization | inaccessible to pilot users | runtime config identity + no invocation | Answer Dispatch / model routing signal only |
| external search | inaccessible to pilot users | pilot account UI/API 强制 `enable_external_search=false` | request scope owner |
| Agent Shadow visible mutation | disabled + contract-gated | changed answer/citations/mode 均为 false | Agent Shadow 仅 auditor |
| citation-bound related assets / source cards | reachable + release-gated | full release gate + headed Portal visibility/permission smoke | final citation visibility owner |
| cross-document advanced comparison | reachable + release-gated | existing comparison/permission release coverage | Answer Dispatch + RetrievalPlan |
| other existing deterministic families | reachable + release-gated | existing focused/release regressions；不得被 pilot flag 改写 | their existing Answer Dispatch executors |

“reachable + release-gated” 表示不进入每次开发都跑的 10 分钟 fast gate，但必须在试点开放前由现有 full release gate 证明。若某项无法被现有 release gate 证明，就必须改为对 pilot 账号不可达，不能只写“未宣称已验证”。

## Constraints

- 不等待完整 Phase 3 后才试点；只关闭当前试点合同之外且无法稳定证明的能力。
- 不引入 LangGraph、LlamaIndex、新 Agent Runtime、新规则引擎或新的 orchestration facade。
- `IntentContract`、`RetrievalPlan`、`EvidenceEnforcementResult`、`AnswerDispatchContract` 与正式 clarification renderer 保持现有 ownership；renderer 只投影 Owner 已提交的 disabled/clarification result，不重新决策。
- 新 gate 只能是 `Validator`，不能写 truth、切换 feature flag、重判 route 或改变用户响应。
- 所有 feature-off 决策由现有 Answer Dispatch Owner 提交；system config 只提供 signal，gate 只验证生效结果。
- `/chat/ask`、`/chat/ask/stream`、auth、department isolation、citation provenance、external/internal source 边界保持稳定。
- 当前 Phase 2 worktree 有并发 writer；本 design doc 不修改其文件。实施前必须重新确认 ownership、diff 与 runtime identity。
- 用户开发效率高，因此不以最少代码为唯一目标；优先最短、最确定的 evidence loop。

## Premises

1. 试点上线不等于 architecture program 全部完成；只要求 enabled pilot contract 被完整证明。
2. “复杂多轮”指用户语义与任务连续性，不要求保留所有内部 fallback、shadow 或 compatibility topology。
3. 临时关闭必须有明确 off variation；不能落入普通生成，也不能在 config 读取失败时静默重开。
4. 快速 gate 与最终 release gate 分工不同：前者支持高频开发，后者仍负责完整 release sign-off。
5. 工艺工程师在 3 分钟内完成一次典型多轮查证，才算产生可见价值。
6. 单个 turn 的 route/citation 通过不能替代完整 conversation 的逐 turn 审阅。

## Landscape

外部官方做法与本设计一致：

- RAG 评估应区分 retrieval 与 retrieve-and-generate，并同时看 correctness、completeness、faithfulness、citation precision 与 citation coverage，而不是只看检索命中。[Amazon Bedrock RAG metrics](https://docs.aws.amazon.com/bedrock/latest/userguide/knowledge-base-evaluation-metrics.html)
- final response 还应单独评估 groundedness、relevance 与 response completeness。[Microsoft Foundry RAG evaluators](https://learn.microsoft.com/en-us/azure/foundry/concepts/evaluation-evaluators/rag-evaluators)
- 多轮系统需要 conversation-level 与 turn-level 两层断言，覆盖 context retention、clarification、信息修正和最终目标完成。[Microsoft multi-turn evaluation](https://learn.microsoft.com/en-us/microsoft-365/copilot/extensibility/evaluation-multi-turn)
- kill switch 的职责是完整关闭功能并提供明确 off variation；它不应把请求送入未声明 fallback。[LaunchDarkly kill switch](https://launchdarkly.com/docs/home/flags/killswitch/)

EUREKA：本仓库不缺 gate，缺的是一个明确的试点 capability view。继续增加评分器、样本 schema 或新的 release truth 只会制造第二 control plane。最小正确动作是组合已有 validators，并让 off behavior 成为 Answer Dispatch 的显式结果。

## Cross-Model Perspective

两次独立只读 repository review 得到一致结论：

- architecture spec 的 baseline 已落后于当前 `HEAD`，Phase 1A 不应重复实施。
- Phase 2 worktree 是 moving target，Agent Shadow auditor-only 已提交，但 runtime acceptance、FA10、permission 与 stream parity 仍未闭环。
- answer-dispatch worktree 已合入 `main`，不应继续 cherry-pick 或开发。
- 最小快速验证应复用 `make verify-retrieval-focused` 与现有 replay/permission/runtime truth，不新增另一个评分或发布系统。
- 当前 named-overview route switch 是 containment switch，不是 artifact 修复；普通生成 fallback 和 fail-open config reader 仍会破坏“真正关闭”的产品语义。

## Approaches Considered

### Approach A: Existing Bundle

改正 named overview fail-closed 语义，但不新增 target；由每个 AI 窗口执行一条固定命令组合。

优点：改动最小，不增加 wrapper。缺点：sample IDs、参数、runtime 与权限步骤容易跨窗口漂移，无法稳定满足用户要求的“一个简单门控”。

### Approach B: Thin Pilot Gate

新增唯一入口 `make verify-production-tech-pilot`，只组合现有 validators，不新增评分逻辑、truth lifecycle、route owner 或 schema。named overview off 由现有 Answer Dispatch 提交决策，formal clarification renderer 返回显式 fail-closed response。

优点：一个命令固定试点合同，适合高频 AI 开发；失败能快速落到已有 test/replay/gate。缺点：仍新增一个 wrapper，必须有 removal trigger，不能永久长成第二 release gate。

### Approach C: Capability Contract

在 `AnswerDispatch` 正式引入统一 capability allowlist/off-policy，替代零散 bool/mode flags，并由 capability contract 驱动 gate。

优点：长期 ownership 最整齐。缺点：会扩大 Phase 1B/Phase 3，当前 moving worktree 上容易再造 V2 owner，不适合短期试点。

## Recommended Approach

选择 **Approach B: Thin Pilot Gate**。

### Decision and Owner

- Pilot capability enable/disable decision：`Answer Dispatch` 是唯一 active Owner。
- `AnswerRouteResolverConfig` / system config：`Signal Provider`，只提供 enabled/disabled input，不能自行写 response 或另选 fallback。
- Disabled-capability final semantic：Answer Dispatch 提交 `capability_disabled` result，正式 clarification renderer 只构造用户可见响应。
- Evidence/no_context：继续由 `EvidenceContract -> EvidenceEnforcementResult` 决定，不由 gate 或 route switch 重算。
- Gate：纯 `Validator`，只消费当前 config、existing tests、existing samples、runtime response 与 permission evidence。
- Final visible write：现有 sync/stream response owner。

执行顺序固定为：Intent Graph 仍识别 named-overview intent，不因 flag 改写意图；resolver config 向 Answer Dispatch 提供 disabled signal；Answer Dispatch 阻断 named-overview executor，并提交 `capability_disabled`；formal clarification renderer 投影 response；gate 只验证结果。

### Named Overview Off Behavior

当 `named_overview_answer_route_enabled=false` 时：

- 不调用 named-overview deterministic builder、composer 或普通 LLM generation。
- 返回明确的用户可见说明，例如：“整篇文档概述功能正在维护，当前可询问具体参数、步骤、检查点或章节内容。”
- 使用现有正式 clarification response shape，`mode=clarification`，`clarification_request.reason=capability_disabled`，`options=[]`，`free_text_allowed=true`。
- 不返回 citations、external_sources 或 related_assets。
- sync 与 stream 的最终 response 等价。
- config 读取失败时保持 capability disabled；不得 fail open。
- 其它明确允许的 deterministic family 不受影响。

该 response 是 stateless clarification variation：不创建待选择的 pending option，不把下一轮当作 option selection。若本轮用户显式指定了可读 document，finalizer 可以保留该 document scope；用户下一轮改问具体步骤、参数或检查点时，应正常进入该文档的 enabled detail flow。必须增加回归：disabled overview -> 具体步骤追问 -> target-document grounded answer。

`capability_disabled` 是用户可见行为语义。实施时应同步登记 `MAIN_CONTRACT_MATRIX.md` 的 clarification behavior boundary；不新增公共字段，但不能只靠测试冻结。

当前 worktree 中“flag=false 后返回普通 grounded RAG”的路径必须在实施前改正；相关测试不能继续用相邻文档 citation 证明所谓安全 fallback。

### Automated Fast Gate Contract

建议入口：

```bash
RETRIEVAL_EVAL_API_BASE=<active-pilot-api> make verify-production-tech-pilot
```

它只组合以下已有真源，并以不超过 10 分钟为目标：

1. `make verify-retrieval-focused ... QUICK_GOVERNANCE=1`，覆盖本轮 route/off behavior、multi-turn scope、response builder 与 stream safety 的 focused tests。
2. runtime source/config identity，证明 API container、worktree checksum、system config path 与 off variation一致。
3. `scripts/eval_chat_memory.py --behavioral` 的 enabled trajectory subset：
   - same-document follow-up；
   - explicit new-document stale-scope reset；
   - external-fallback-disabled negative。
4. formal ambiguity clarification 的现有 tests，以及一个 bounded live conversation：ambiguous title -> option -> selected-document detail。
5. named overview disabled negative：明确 fail-closed、无 generation、无 citation、无 hidden fallback。
6. 四个本地测试账号的 API replay，验证 department/document visibility。
7. 本试点使用 Portal streaming，必须成对比较 sync/stream 的 final answer、mode、citation identity 与 clarification。
8. 对 enabled turns 使用现有 deterministic verifier / expected facts / forbidden markers 自动检查 visible body，不只检查 status、route 或 citation count。

该 target 不新增新的持久化 report schema。各 constituent runner 继续输出既有 artifact；gate 只汇总退出码与简短终端结果。

### Gate Inputs and Fail Conditions

固定输入：

- `RETRIEVAL_EVAL_API_BASE`：指向当前 pilot branch runtime。
- `.branch-runtime.local`：提供生效的 worktree/container/port/config identity，以及四组本地测试账号变量；只检查变量存在性，不输出值。
- enabled sample IDs：`memory-linear-guide-inside-checkpoints-followup-001`、`memory-gear-sop-clears-stale-linear-guide-scope-20260530`、`memory-external-fallback-disabled-001`。
- 新增普通 regression row `pilot-production-tech-ambiguous-title-selection-001`，只使用现有 `eval/chat_memory_samples.yaml` schema，覆盖 ambiguous title -> option -> selected detail。
- 新增普通 regression row `pilot-production-tech-named-overview-disabled-001`，只使用现有 schema，覆盖 explicit named overview -> `capability_disabled` -> specific-detail follow-up。
- auth aliases：现有 `PORTAL_SMOKE_USERNAME` / `PASSWORD` 及 `_2`、`_3`、`_4` 变量；不得把值写入 tracked artifact。

以下任一情况必须返回非零，不允许 `skip` 后继续判绿：

- API/health 不可达，或 runtime worktree/container/config source/checksum 与预期不一致。
- required env/account 变量缺失。
- sample selector 返回 0、数量不等于固定 cohort，或 required row 被 status/track 过滤。
- 任一 focused test、behavioral turn、permission check 或 sync/stream pair 未执行。
- named overview disabled path调用任何 deterministic/composer/ordinary generation，或返回 citation/external source/related asset。
- enabled answer turn 缺少具体事实、使用错误 document citation、出现 forbidden marker/template，或被记录为 `false_pass_visible_answer_body`。
- clarification selection 前提前锁定 scope，或显式 new document 未清除 stale scope。
- stream 已启用但 final semantic parity 未检查。

fast gate 的 visible-body 检查必须自动化为 existing sample expectations / verifier；任何无法自动表达的正文质量判断不塞进该命令，而进入下面的人工 acceptance。

### Manual Pilot Acceptance

在试点开放前单独执行，不受 10 分钟目标约束：

1. 对 fast gate 与 full release artifacts 做一次完整 visible-body/citation 人工审阅并显式记录通过/阻塞结论。
2. 运行 `make verify-retrieval-release`。
3. 使用四账号完成 headed Portal smoke，核对登录、stream、source cards、related assets、permission 与 disabled capability 文案。
4. 在 gate 通过后的 pilot runtime 上，完成一位工艺工程师的不受引导三轮 acceptance observation。

不新增 attestation schema；结论写入当前 implementation final/handoff 即可。没有人工审阅或 headed smoke 时只能说 fast gate green，不能说 pilot accepted。

### Gate Boundaries

- 自动 fast gate 目标完成时间：不超过 10 分钟；首次稳定前记录实际耗时，不为追求速度降低验证语义。
- 每个 user-visible chat change 必须运行。
- 本试点明确使用 Portal streaming，因此 Phase 2B finalized stream metadata parity 是无条件硬阻塞项，不再按 sync-only 例外处理。
- 最终试点上线仍需完成 Manual Pilot Acceptance。
- FA10 overview residual 继续保留为未修复证据；在 capability off 时，它作为 disabled negative，而不是 positive overview pass。
- Phase 2A 的 Shadow runtime proof改用 enabled multi-turn trajectory，不再要求一个被明确关闭的 overview capability产生正向答案。

### Complexity and Exit

本方案增加一个临时 Make wrapper和一个明确 off behavior，但从试点 active topology 中删除 named-overview deterministic/composer/generic fallback，active Owner 数量不增加，运行拓扑净减少。

```text
[REMOVAL_CANDIDATE] Makefile::verify-production-tech-pilot
  reason: temporary pilot validator wrapper; existing release gates remain authoritative
  owner: existing focused/release validators
  real_consumer: production technology pilot development loop
  trigger: after pilot graduation, when enabled trajectories are absorbed by existing release cohorts and no pilot-only off capability remains
  last_verification: pending first implementation and live pilot gate
```

```text
[REMOVAL_CANDIDATE] AnswerRouteResolverConfig.named_overview_answer_route_enabled
  reason: temporary emergency kill switch for an unreliable overview route
  owner: Answer Dispatch without a permanent parallel boolean
  real_consumer: current pilot off behavior
  trigger: old overview route is deleted, or artifact-backed overview passes fresh live replay plus provider-capacity proof and is intentionally re-enabled
  last_verification: current moving worktree implementation is uncommitted and not yet runtime-proven
```

## Open Questions

- 第一位工艺工程师及首批真实文档尚未在本设计中命名；实施 gate 时应使用已获授权的真实 pilot corpus，不把 synthetic fixture 当最终证据。
- 当前 permission visibility 的 6 条 ACL-drift rows 需要重新归因并闭环，不能因 retrieval-regression count 为 0 就跳过。
- formal ambiguity live trajectory 如没有 checked-in sample，可新增普通 regression row；不得创建新的 eval 字段或 truth lifecycle。
- named overview 的具体提示文案和 clarification options 需要一次 Portal 可见性检查，但不应扩大为新的 UX 项目。
- moving worktree 的并发 writer 必须先完成、提交或明确移交 ownership；不能直接覆盖当前 dirty files。

## Success Criteria

### Product

- 一位工艺工程师在不受引导的情况下，于 3 分钟内完成一次典型三轮查证。
- 相比 10–20 分钟人工查找，至少节省一半时间，并且用户能指出他实际核对了哪些来源。
- 用户遇到 disabled named overview 时能理解下一步可问“参数、步骤、检查点或章节”，而不是误以为系统坏掉。

### Behavior

- enabled pilot trajectories 100% 通过，逐 turn scope、answer body、citation 与 clarification 无 material failure。
- permission leak = 0。
- citation document mismatch = 0。
- false_pass_visible_answer_body = 0。
- stale-scope inheritance failure = 0。
- ambiguity false certainty = 0。
- disabled capability hidden fallback = 0。
- sync/stream final semantic mismatch = 0。

### Engineering

- `make verify-production-tech-pilot` 在目标 runtime 上一次命令完成，目标耗时不超过 10 分钟。
- gate 只复用 existing validators/samples，不新增评分 truth、eval schema 或第二 route owner。
- Phase 2B 在 Portal stream 上线前完成并通过 paired replay。
- 最终试点上线前 `make verify-retrieval-release`、四账号 permission proof、headed Portal smoke 与完整 visible-body review 全部通过。
- `MAIN_CONTRACT_MATRIX.md` 在实施时完成双向检查；若增加稳定的 disabled clarification reason/behavior，应同步登记，不能只写测试。

## Distribution Plan

产品仍通过现有 Portal/web service 与当前部署流水线提供，不新增二进制、包或独立服务。

试点 gate 通过后，只向生产技术部门的授权账号开放。named overview kill switch 保持 off；任何重新启用都必须先满足 removal trigger，不允许通过临时 env 或共享 config 手工绕过。

## Dependencies

- 当前 Phase 2 worktree ownership 与 dirty diff 收口。
- named overview switch 的 off behavior 从 ordinary generation fallback 改为 formal fail-closed。
- Phase 2B finalized stream metadata parity；本试点使用 Portal streaming，因此是硬依赖。
- runtime source/config identity proof。
- 四账号 permission/ACL drift 闭环。
- 至少一个真实工艺工程师和授权 pilot corpus。
- 最终 release gate 与 Portal headed smoke。

Provider capacity、section-index promotion与 FA10 artifact repair是 named overview重新启用依赖；在该能力明确关闭时，不应继续阻塞其它 enabled pilot capabilities。

## Implementation Handoff

### Owner and Order

1. 当前 Phase 2 worktree writer 先提交、移交或明确放弃 dirty ownership；新 executor 不得覆盖现有 15 个 tracked dirty files 与 untracked plans。
2. Answer Dispatch owner 修正 named-overview off path：Signal Provider -> unique Owner -> stateless clarification renderer，并补 `MAIN_CONTRACT_MATRIX.md`。
3. 完成 Phase 2B finalized stream metadata parity，证明 Portal transport 不拥有第二套 final semantics。
4. 添加 validator-only `verify-production-tech-pilot` wrapper、两条普通 regression rows 与固定 fail-closed inputs。
5. 关闭 permission/ACL drift 与 runtime identity blockers，运行 automated fast gate。
6. 运行 full release、四账号 headed Portal smoke 和 gate 后的真实工艺工程师 acceptance observation，再决定 pilot acceptance。

### Hard Blockers Before Implementation

- moving worktree ownership 未收口。
- disabled signal 到 Answer Dispatch 的 existing seam 未在计划中明确。
- `capability_disabled` response、stateless clarification 和下一轮 scope 回归未设计进 tests。

### Hard Blockers Before Pilot Acceptance

- Phase 2B 未完成。
- permission/ACL drift 与 runtime config identity 未闭环。

### Reactivation-Only Blockers

provider capacity、section-index promotion、FA10 267-section artifact 与 overview composer readiness 只阻塞 named overview 重新启用；在 fail-closed off contract 通过后，不阻塞其它 enabled pilot capabilities。

### Ready for Plan

只有在 worktree ownership 清楚、pilot corpus 与四账号变量可用、两条 planned regression rows 的真实文档/期望事实已确定后，才进入 bounded implementation plan。计划不得同时扩张 Phase 1B/Phase 3 或新增 capability framework。

## The Assignment

先安排一位工艺工程师做一次 pre-plan discovery observation：用代表性授权文档走完三轮查证，第一轮问明确文档中的具体要求，第二轮省略文档名继续追问，第三轮显式切换到另一份文档。记录总耗时、每轮是否保持正确 document scope、他实际点开核对了哪些 citation，以及何时想退回原文。该观察可以使用当前可用 runtime，目的是校准任务、文案和 regression inputs，不构成 pilot acceptance。

pre-plan discovery observation 必须发生在写实施计划之前或并行完成。实现与 gate 通过后，再以 pilot runtime 重复一次不受引导的 post-gate acceptance observation；后者才进入 `Manual Pilot Acceptance`。两次观察都是需求与 UX 证据，不由自动 gate 替代。

## What I noticed about how you think

- 你说“我的开发效率很高”，这把真正瓶颈从写代码移到了证明代码。设计因此选择一个稳定验证入口，而不是继续压缩实现范围。
- 你没有接受我建议的“单文档事实问答”，而是明确选择“复杂多轮问答”。这说明你愿意砍内部拓扑，但不愿砍用户真正需要的上下文连续性。
- 你同时说“短期内上线”和“暂时关闭部分功能”。这不是追求功能数量，而是在主动给试点划可信边界。
- 你要求“新增一个简单门控”，而不是再写一份完整 SOP。这个偏好直接排除了新的评分系统、campaign artifact或第二套 release truth。
