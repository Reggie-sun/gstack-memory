# Design: Internal Launch Safety-First RAG Guard

## Problem Statement

当前项目需要在 2 天内做一次内部上线，但最新 retrieval 证据显示准确率和引用稳定性都不够安全。对这次上线来说，真正的风险不是“答不上来”，而是系统给出一个看起来很完整、还带着引用的错误答案，导致部门流程负责人被误导并失去信任。

这次要解决的不是“把系统整体变聪明”，而是优先把高风险错答和错引压下去，让系统先具备可信的上线姿态。

## Demand Evidence

- 当前最强需求证据不是已有用户离不开它，而是明确的内部上线节点。
- 用户明确把上线标准收敛为：“至少别答错并能给对引用。”
- 当前仍处于 pre-product 阶段，说明这次最重要的是守住内部首次试运行的信任，而不是追求全量覆盖。

这说明当前任务本质上是一个内部交付与信任控制问题，不是外部市场需求验证问题。

## Status Quo

- 当前系统答不准或引用不对时，内部最真实的替代方案仍然是人工回查原文。
- 这意味着当前系统的真实竞争对象不是别的 AI，而是“共享盘/文档系统/同事协助”的手工查证流程。
- 如果系统不能做到“比人工更快且不把人带偏”，用户会自然回退到人工原文检索。

## Target User & Narrowest Wedge

目标用户：
- 部门流程负责人

他们最怕的不是系统保守，而是：
- 把多个制度、流程、工艺口径拼错
- 给出看似有依据的错误引用
- 让用户在不知情的情况下沿错误答案继续推进工作

最小切口：
- 第一版不追全场景提准
- 先把“有证据才展示答案和引用”作为上线主规则
- 首发目标不是“多答”，而是“别错答、别错引”
- 优先保住部门流程负责人最典型的问题路径，而不是追求 broad semantic coverage

## Constraints

- 只有 2 天窗口，不适合大规模重构 retrieval 主逻辑。
- 当前仓库已经有 `no_context` / `retrieval_fallback` / citation / grounding / rewrite 这些可复用护栏，优先复用而不是重造。
- 现有证据显示：
  - frozen retrieval gate 曾达到 `37/37`
  - 最新 `2026-05-15` eval 已下降到 `top1_accuracy=56.6%`、`topk_recall=62.3%`
  - `structured_subsets.phase2a_canary` 仅 `33.3%`
- 这说明 2 天内不应押注“把整条 retrieval 主线重新拉回 100%”，而应优先压缩错误外露面。
- 当前人工回查仍是默认 fallback，因此“更保守”是可接受的，“更自信地错”不可接受。
- 本轮 `plan-eng-review` 已确认第一版必须收窄到最小安全补丁，避免在 2 天窗口内扩成 `8+` 文件的中型 diff。

## What Already Exists

- `backend/app/services/chat_service.py`
  - 已有 `_build_response()`，会解析 `used_context_numbers` 并过滤可见 sources
  - 已有 `no_context` / `retrieval_fallback` 主响应模式
- `backend/app/services/chat_source_grounding.py`
  - 已有 `filter_chat_sources_by_context_indexes()`，能把模型声明使用的上下文编号映射回 citation / external source
- `backend/app/services/chat_anchored_snippet_answering.py`
  - 已有 deterministic anchored-snippet 路径，可优先复用于“少总结、多贴证据”的首发问法
- `backend/app/rag/generators/client.py`
  - 已有回答 system prompt 与 metadata marker 协议，但当前 wording 仍偏向“证据不精确也可给一般说明”
- `backend/tests/test_chat_source_grounding.py`
  - 已有 source grounding 与 metadata 过滤基础测试
- `backend/tests/test_retrieval_chat.py`
  - 已有 `no_context` / `retrieval_fallback` / external fallback 的主链路测试

## Review Decisions

- 第一版范围收窄到约 `5` 个核心文件，不同时动 rewrite、citation ranking 和更广的 eval 体系。
- 增加一个 **backend-private rollout gate**，只给服务端回切用，不进入 public contract。
- 当 `mode=rag` 且要求 `used_context_numbers` 时，如果 grounding 过滤后 internal citations 为空，升级成 **response-level safety gate**，不再返回普通 `rag answer`。
- 新 safety rule 必须集中到一个共享 helper，不允许在 sync / stream / fallback 分支各写一份。
- internal safety gate 与 external fallback **默认解耦**：internal 证据不足时先安全降级，不自动外补。
- 第一版测试必须同时锁住 sync / stream / external-fallback 的行为矩阵，以及 malformed metadata 脏输出用例。

## Premises

1. 这次内部上线里，错答加错引比答不上来更糟。  
2. 2 天内应该优先做 evidence sufficiency 护栏、abstain/降级和引用过滤，而不是先追求更高召回。  
3. 首发范围应先收窄到部门流程负责人最典型的场景，而不是追全量覆盖。  

以上前提已在本次 office-hours 中确认。

## Approaches Considered

### Approach A: Safety-First Launch Guard

在现有 chat / retrieval / citation 主线上增加更严格的“证据充分性”控制：

- LLM 未明确声明使用哪些上下文时，不展示 citation
- 引用与 answer context index 对不上时，不展示 citation
- 证据不足时优先回落到 `no_context`、`retrieval_fallback` 或更保守回答
- 必要时对高风险 broad/semantic 问题直接收紧回答模式

优点：
- 2 天内最容易落地
- 直接命中“别答错并给对引用”
- 最大化复用现有护栏与测试资产

缺点：
- 覆盖率会下降
- 用户短期会感知系统更保守

### Approach B: Targeted Retrieval + Citation Repair

针对部门流程负责人最常见场景，集中修 retrieval 命中、query rewrite、same-doc citation 选择与排序。

优点：
- 有机会同时提升正确率和可答率
- 产品观感更像“变好了”

缺点：
- 2 天内容易扩散成打地鼠
- 如果根因在 corpus drift、citation 对齐或多层联动，不一定能稳定收口

### Approach C: Whitelist Wedge

首发只开放一小批高价值、高可控文档族与问题类型，尽量走 extractive / anchored-snippet 答法。

优点：
- 最快建立信任
- 容易对人工回查形成明确优势

缺点：
- 范围收窄明显
- 产品形态会更像“受限首发”

## Recommended Approach

推荐 Approach A: Safety-First Launch Guard。

理由很直接：

- 这次上线目标是“别答错并给对引用”，不是“尽量多答”。
- 外部研究与竞赛实践都指向同一方向：当证据不足时，正确的行为是 `abstain` 或丢弃不可靠引用，而不是继续生成貌似完整的答案。
- 当前仓库已有可复用护栏，包括：
  - `backend/app/services/chat_source_grounding.py`
  - `backend/app/services/chat_citation_pipeline.py`
  - `backend/app/services/chat_anchored_snippet_answering.py`
  - `mode=no_context / retrieval_fallback`
  - rewrite / telemetry / request snapshots / retrieval eval corpus
- 这条路可以在不大改 retrieval 架构的前提下，最快把“危险答案”挡在上线前。

建议的第一版形态：

1. 把“是否展示答案与引用”从“检索到了候选就展示”改成“证据足够且对齐才展示”
2. 对缺失 `used_context_numbers`、无有效 grounded citation、上下文索引越界、answer 没有真正使用 citation 的情况，默认隐藏 citation；若 internal grounded citations 为空，则升级为 response-level safety gate
3. 增加一个 **backend-private rollout gate**，用于上线后快速回切 Safety-First 行为，不扩 public contract
4. 新 safety rule 集中在一个共享 helper，不引入新的 orchestrator service
5. internal safety gate 默认不自动触发 external fallback；外部补答仍由显式联网意图或后续单独策略决定
6. 对高风险 broad/semantic 问题增加更保守的拒答或降级策略
7. 对首发重点问法尽量走 anchored snippet / same-doc evidence 优先，而不是自由总结
8. 上线评估不看全量“能不能答”，优先看：
   - 错答率
   - 错引率
   - 高风险场景是否能被安全挡住

## Open Questions

- 哪几类问题必须答，哪几类问题宁可直接拒答？
- 是否存在一批可接受的首发文档白名单，能进一步辅助 Safety-First 策略？
- “给对引用”的验收口径是 document-level 正确，还是 snippet/chunk-level 也必须正确？
- backend-private rollout gate 采用哪一种最小实现更合适：内部 settings、env toggle，还是服务端私有配置分支？

## Success Criteria

- 对部门流程负责人最典型问题，系统不再输出“明显错答但看起来有引用支撑”的结果
- citation 只在 answer 真正使用且能对齐上下文时展示
- 高风险低证据问题优先进入 `no_context` / `retrieval_fallback` / 更保守回答
- 上线前至少有一组关键问集能证明：
  - 错答率下降
  - 错引率下降
  - 首发重点问题仍可用
- 对最新 eval，不再把“更多能答”当成唯一指标，而把“安全输出比例”纳入主门槛

## Distribution Plan

本方案面向现有 Web 服务，不新增新的独立交付物。交付路径沿用当前 API 与前端发布链路。

上线方式建议：
- 先在内部环境用关键问集做 focused gate
- 先以 Safety-First 配置首发
- 首发后再决定是否逐步放开更激进的 retrieval / semantic coverage

## NOT in scope

- `query_rewrite_service` 调整
  - 这轮先不碰。理由：2 天窗口里它更像准确率提升项，不是最小安全补丁必需项。
- `chat_citation_pipeline` ranking / same-doc reorder 深修
  - 这轮先不碰。理由：风险高、测试面广，容易把补丁扩成 retrieval 行为改造。
- `chat_anchored_snippet_answering` 新策略扩展
  - 这轮先不碰。理由：可复用现有能力，但不把它变成首刀主战场。
- public `system_config` / schema / frontend 配置面
  - 这轮先不碰。理由：rollout gate 只做 backend-private，避免 contract 扩散。
- 全量 retrieval eval 重建或 accuracy 主线重拉
  - 这轮先不碰。理由：这次目标是控错答风险，不是把全链路指标重新拉回历史峰值。

## The Assignment

先不要想着“把全部准确率拉上去”。先做一件更值钱的事：

列出 10 到 20 个部门流程负责人最关键的问题，逐条标记：

1. 哪些问题“必须答且必须给对引用”
2. 哪些问题“答不上来也可以，但绝不能错答”
3. 哪些问题“当前可以直接禁止回答或降级”

如果这张清单做不出来，后面的护栏实现很容易又退回“全都想答”的老路。

## What I noticed about how you think

- 你没有把这次问题描述成“准确率低”，而是直接收敛成“至少别答错并能给对引用”。这说明你关心的是信任边界，不是好看的指标。
- 你在只有 2 天窗口时，没有要求“整体重做 retrieval”，而是优先找最快能守住上线的方法。这是很强的交付判断。
- 你愿意接受范围和野心收缩，只要能换来更可靠的首发体验。这比一味追求全能更像真正的内部产品负责人思路。
