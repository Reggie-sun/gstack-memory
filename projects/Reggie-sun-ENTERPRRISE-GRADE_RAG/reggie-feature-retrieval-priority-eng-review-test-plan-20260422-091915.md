## Affected Pages/Routes
- `POST /plugin-runtime/agent-runtime` — verify `resolve-workflow` and `advance-workflow` preserve structured runtime semantics and error payloads.
- `GET /plugin-runtime/schema` — verify schema stays aligned with supported commands.
- `runWorkflowFromPrompt(prompt, changedPaths?, taskPath?)` — verify host-side prompt routing and auto-advance behavior.

## Key Interactions to Verify
- Prompt asking for next step should call `resolve-workflow` only.
- Prompt asking for direct execution should call `resolve-workflow`, then `advance-workflow` only when runtime explicitly returns `auto_execute_recommended`.
- `workflow.advance` should surface the executed capability's real `ui_intent` and `terminal_state`, not a flattened wrapper value.

## Edge Cases
- Bridge returns non-JSON error payload (HTML/plain text/proxy error) and SDK still raises a stable actionable error.
- Manual-routing / inspect path goes through `workflow.advance` without being mislabeled as `review_execution`.
- Missing `changedPaths` falls back cleanly to repo-side detection.

## Critical Paths
- High-risk change with no task contract: `resolve-workflow` blocks with `collect_task_contract`, `advance-workflow` returns preflight blocker.
- Safe API change with valid task contract: `resolve-workflow` recommends `flow.execute`, `advance-workflow` executes the concrete lane.
- Direct host flow: `runWorkflowFromPrompt` -> `resolve-workflow` -> conditional `advance-workflow`.
