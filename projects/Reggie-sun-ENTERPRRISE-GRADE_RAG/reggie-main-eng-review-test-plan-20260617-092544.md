## Affected Pages/Routes

- `/api/v1/chat/ask` - 验证目录直达、检索路径、文件名直达在 sync chat 中按同一 RouteDecision 行为分流。
- `/api/v1/chat/ask/stream` - 验证 streaming 与 sync 对同一目录直达问题保持同样的 direct listing、diagnostics 和跳过 LLM 行为。
- Internal retrieval path - 验证 `RetrievalService.search_candidates()` 继续只消费已有 metadata result/request scope，不因为 RouteDecision 新增 folder metadata 读取。

## Key Interactions to Verify

- 用户问“质量体系目录下有哪些文件”时，返回 metadata listing，跳过 retrieval pipeline、rerank 和 LLM，并写入 `route_decision_action=direct_listing`。
- 用户问“CRM通用操作文档V1.3版主要讲什么”时，不走目录直达，进入检索路径，并写入 `route_decision_action=retrieval_scoped_answer`。
- 用户问“找一下 <filename> 这个文档”时，保持现有 deterministic document lookup answer，同时保留 `route_decision_goal=document_lookup` 一类诊断。
- 单个目录直达结果后的追问仍能通过旧 `metadata_anchor_*` memory fields 继承正确目录或文档范围。

## Edge Cases

- RouteDecision 输入没有 folder summaries、只有 document `category_id` 时，retrieval scoped recall 仍能基于 category paths 工作。
- `metadata_anchor_*` legacy diagnostics 和新增 `route_decision_*` diagnostics 同时存在，不破坏 telemetry、no_context diagnostics 和 follow-up scope。
- document title/content 问法必须 defer 到 retrieval，不能因为 listing hints 或 “找下/找一下” 回到目录直达。
- category scope/request scope 与 query inferred scope 冲突时，RouteDecision 要暴露 reason，现有更具体 query anchor 优先逻辑不能退化。

## Critical Paths

- Directory direct: `ChatAnswerHandler` -> metadata anchor listing -> deterministic `ChatResponse` -> memory/telemetry.
- Retrieval path: `ChatCitationPipeline` -> `RetrievalService.search_candidates()` -> metadata anchor scoped recall -> vector/hybrid candidates -> deterministic/LLM answer.
- Compat path: RouteDecision diagnostics added without replacing `metadata_anchor_intent`, `metadata_anchor_matched_folder_paths`, or `metadata_anchor_listed_folder_paths`.
