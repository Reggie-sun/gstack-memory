## Problem Statement

系统 OCR/screening 优化已经做了 Phase 2B cleaning audit，lite screening 已落地，检索 gate 37/37 全绿。问题是：**继续打磨 OCR，还是先让真实用户用上系统？**

答案：不应该继续优化 OCR。应该先上线。

## Demand Evidence

- 内部有具体的人现在每天花时间翻纸质文件、问同事来查 SOP 和故障代码
- 系统检索能力已达到 37/37 = 100% gate
- 系统还处于 prototype 阶段，没有真实用户
- OCR 噪音的影响在当前 eval 样本范围内不可见，但真实用户的 query 模式可能不同

## Status Quo

目标用户现在的"解法"：翻纸质文件、翻文件夹里的 PDF、问老同事。慢、容易出错、依赖老员工在岗。

## Target User & Narrowest Wedge

### 用户画像

**用户 A：生产线班组长**
- 角色：负责生产线日常运行，遇到设备报警时需要快速查故障处理流程
- 技术水平：会用电脑，但不擅长复杂软件操作
- 设备：工位上有共享电脑，内网访问
- 语言：中文，习惯用"故障代码 E-003"、"A05 操作规程"这样的简短查询
- 典型场景：设备报警后需要 5 分钟内找到处理步骤，现在翻纸质文件要 10-20 分钟

### 最小楔子

- 最小可用版：一个搜索入口 + 返回结果 + 反馈按钮
- 不需要多轮对话、不需要权限细分、不需要完美的 OCR 清理
- 搜索框输入故障代码或 SOP 名称，返回文档名 + 页码 + 关键段落

## Constraints

- 不做 breaking changes 到现有 API（但允许 additive changes，如新增 `query_id` 字段）
- 不碰 OCR/screening（当前 lite 是 working baseline）
- 不做通用化给所有人
- 沿用现有 FastAPI + React/Vite 技术栈

## Premises

1. OCR 噪音在当前 37 条样本范围内不是瓶颈，但真实用户可能暴露不同模式。**更准确的表述：还没有证据证明它是瓶颈，也没有证据证明它不是。**
2. 最大瓶颈是让真实用户用上系统。能说出具体的人和场景 = 有真实需求。
3. OCR 噪音的影响只有真实使用才能准确判断。request_snapshot + event_log 可以收集证据。

## Cross-Model Perspective

Codex (gpt-5.4) 独立审查，三个关键观点：

1. **最强版本不是更干净的 OCR 管线，而是给那个具体用户的"SOP/故障码查找台"。** 基于已过 gate 的 retrieval，把"翻文件+问同事"替换成"输入问题 -> 拿到原文片段 -> 反馈"。

2. **能说出具体的人 = 有使用楔子。** 缺的不是算法方向，是最短可用路径。

3. **前提 1 的修正：** 更准确的说法是"没有证据证明 OCR 是瓶颈，也没有证据证明它不是"。验证方式：上线一周后按文档类型分层看失败 query。如果集中在扫描件/OCR 重文档，OCR 就是隐藏瓶颈。

4. **48 小时原型建议：** 单页搜索界面 + top 5 结果（文档名/页码/摘要/关键词） + 3 个反馈按钮（找到了/没找到/结果不准） + 极简失败看板。跳过 OCR、复杂 rerank、权限细分、花哨 chat UI。

## Approaches Considered

### Approach A: 先上线，后优化

- Summary: 直接部署当前版本给目标用户，收集反馈
- Effort: S
- Risk: Low
- Pros: 最快让用户用上；真实数据驱动后续优化
- Cons: 没有监控机制，不知道用户遇到什么问题
- Reuses: 现有 PortalChatPage、chat API、retrieval 服务
- Completeness: 7/10

### Approach B: 先搞定 OCR，再上线

- Summary: 继续推进 graded screening，处理 PPTX glyph 和 DOCX 表格残留
- Effort: M
- Risk: Med (优化方向可能不对)
- Pros: 系统更干净，演示效果更好
- Cons: 用户要等；可能优化了不重要的方向；37/37 gate 已经够好
- Reuses: 现有 PreStockScreeningService
- Completeness: 9/10

### Approach C: 最小可用版 + 反馈机制 (Recommended)

- Summary: 部署给用户 + 加反馈按钮 + 用现有 request_snapshot 收集真实 query + 手动周回顾
- Effort: M
- Risk: Low
- Pros: 上线快 + 有数据驱动 + 反馈机制简单可实现
- Cons: 需要加反馈 UI 和 API（明确的工作量）
- Reuses: request_snapshot_service、event_logs、PortalChatPage、eval 基础设施
- Completeness: 8/10

## Recommended Approach

**方案 C：最小可用版 + 反馈机制。**

理由：
- 你已经有 request_snapshot 和 event_log 基础设施
- 让真实用户用起来比任何优化都重要
- 反馈机制让 OCR 是否是瓶颈由数据回答，而不是猜测
- 看板在单用户试点阶段是 YAGNI，手动周回顾足够

### 具体实现步骤

#### Step 0 (前置): 确认部署环境和用户访问

- **假设**: 目标用户可以通过内网访问现有 Docker Compose 部署
- **动作**: 确认用户工位电脑能否访问部署 URL，如需 VPN 或防火墙配置则先处理
- **fallback**: 如网络不通，优先解决网络问题再开始其他工作

#### Step 1: 加 query_id 到 ChatResponse (additive change)

现有 `ChatResponse` 没有 query_id，反馈时无法关联原始请求。

- **数据模型**: 在 `ChatResponse` schema 加 `query_id: str` 字段（UUID 格式）
- **生成逻辑**: 在 `ask_question` endpoint 生成 query_id 并写入 request_snapshot
- **API 变化**: 返回的 response 多一个字段，前端无需改动即可兼容

#### Step 2: 创建反馈 API

- **Endpoint**: `POST /chat/feedback`
- **Request body**:
  ```python
  {
    "query_id": str,           # 关联到原始 ChatResponse
    "feedback_type": str,      # enum: "found" | "not_found" | "inaccurate"
    "top_citation_ids": list[str],  # 用户看到的前几个结果 ID
    "comment": str | None      # 可选的额外说明
  }
  ```
- **存储**: 写入 `data/feedback_logs/YYYY-MM-DD.jsonl`，格式：
  ```json
  {"query_id": "...", "feedback_type": "not_found", "top_citation_ids": [...], "ts": "2026-04-11T10:00:00Z", "comment": null}
  ```
- **认证**:沿用现有 `get_current_auth_context`，需要登录

#### Step 3: 前端加反馈按钮

在 PortalChatPage 的每个回答下方加 3 个按钮：
- **找到了** → `feedback_type: "found"`
- **没找到** → `feedback_type: "not_found"`
- **结果不准** → `feedback_type: "inaccurate"`

点击后调用 `/chat/feedback`，按钮状态变为已提交（不可重复点击）。

#### Step 4: 为目标用户创建账户

- **动作**: 在系统中创建用户账号，验证登录流程
- **权限**: 给与目标用户访问文档的部门权限（沿用现有 ACL）

#### Step 5: 部署并让用户试用

- 部署到内网 URL
- 给用户账号和访问方式
- 第一天陪同使用，记录首次体验问题

#### Step 6: 手动周回顾

- **方法**: 每周五 grep `data/feedback_logs/*.jsonl` 和 `data/request_snapshots/*.jsonl`
- **关注点**:
  - `not_found` 反馈占比
  - `inaccurate` 反馈的 query 和结果
  - 零结果 query 的文档类型分布
- **决策阈值**: 如果 `not_found` > 30% 且集中在扫描件/OCR 重文档，OCR 升级为下一优先级

## Decision Gate: OCR 是否是瓶颈

**一周后数据判断标准：**

| 条件 | 结论 |
|------|------|
| `not_found` 反馈 > 30%，且失败 query 涉及的文档中扫描件占比 > 50% | OCR 是瓶颈，下一阶段做 graded screening |
| `not_found` < 10%，但 `inaccurate` > 20% | 检索精度问题，回到 retrieval 优化 |
| 反馈量少（< 5 条），但用户使用量也少（< 20 次查询/周） | 用户没真的用起来，先解决 adoption |
| 用户使用量正常（> 50 次查询/周），反馈多数是 `found` | 系统已足够好，不需要继续优化 OCR |

## Success Criteria

- 目标用户在一个工作日内至少使用系统 5 次查 SOP/故障代码
- 系统记录真实 query 和反馈（> 20 条反馈/周）
- 一周后有数据回答"OCR 噪音是不是瓶颈"
- 用户反馈"找到了"占比 > 60%

## Distribution Plan

- 内部部署，沿用现有 Docker Compose 方案
- 给目标用户内网 URL + 账号密码

## Dependencies

- Step 0 完成：用户能访问部署 URL（内网连通）
- 目标用户有时间试用（协调好首次使用时间）

## The Assignment

**这周内把系统部署给那个具体的人。** 不是下周，不是"等 OCR 处理完"。是这周。问他一个问题："你今天查了什么 SOP？"如果他用你的系统查到了，你就赢了。如果他没查到，你就有了真实数据知道该修什么。

具体执行：
1. 周一：确认网络连通 + 创建用户账号
2. 二：加 query_id 和反馈 API（后端）
3. 周三：加反馈按钮（前端）
4. 周四：部署 + 陪同首次使用
5. 周五：回顾本周数据，决定下一步

## What I noticed about how you think

- 你问"OCR 应不应该继续优化"而不是"我该怎么让用户用上"，说明你的直觉已经指向正确方向了，只是需要外部确认。你自己知道 OCR 不是当务之急，但你已经投入了很多精力在 retrieval 优化上，停下来需要勇气。
- 你能说出具体的人和场景，这在内部项目中很罕见。大多数人只会说"工厂的人需要"。你说的更具体，这意味着你对用户有真实的了解。
- 你选了方案 C 而不是方案 B。在已经做了大量 retrieval 优化之后，选择"先上线再说"而不是"再优化一点"，这是好的产品判断。
