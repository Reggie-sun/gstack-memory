# Design: 表格结构化 — 全格式表格提取与感知 Chunking

Generated by /office-hours on 2026-04-11
Branch: feature/retrieval-priority
Repo: Enterprise-grade_RAG
Status: DRAFT
Mode: Builder

## Problem Statement

RAG 系统无法正确检索表格内容。PDF 完全没有表格提取能力，DOCX/PPTX/XLSX 的表格提取只保留 tab 分隔的纯文本，丢失行列关系。表格被当作普通文本按字符切割，可能被拦腰截断。用户在问参数、规格、检验标准等表格类问题时得不到正确答案。

## 影响评估

### 评估发现

1. **Eval 样本中 4 条表格相关查询全部 disabled**：`fine-002`、`cross-007`、`lexical-002`、`cross-015` 均指向电镀检验标准文档，因表格解析失败导致文档被删除
2. **当前 verified-only baseline 不包含表格场景**：所有 verified 样本都避开了表格场景
3. **剩余参数查询能工作纯属偶然**：参数值碰巧出现在 SOP/WI 的正文文本流中，而不是在表格里

### 当前能力

| 格式 | 表格提取 | 结构保留 | 感知 Chunking |
|------|----------|----------|---------------|
| PDF | 无 | 无 | 无 |
| DOCX | tab 分隔文本（保留 cell 内容，丢失行列关系） | 无 | 无 |
| PPTX | 扁平文本（cell 文本混入普通段落） | 无 | 无 |
| XLSX | tab 分隔文本（保留 cell 内容，丢失行列关系） | 无 | 无 |
| CSV | tab 分隔文本 | 无 | 无 |

## Constraints

1. 这是 ingestion-side 问题，不修改 retrieval 主逻辑。`chunk_type: "table"` 作为 metadata 存入 Qdrant payload，本阶段不影响检索评分
2. 不改变现有 public API 契约（retrieval/search, chat/ask）
3. 全量重建索引需要回滚预案（已有 OPS_CHUNK_REBUILD_RUNBOOK.md）
4. 必须能在现有 eval baseline 上做 before/after 对比
5. 与现有 retrieval optimization plan 不冲突，表格结构化作为 Phase 2B 的一个子工作流插入

## Premises

1. 表格结构化是 ingestion-side 问题，不是 retrieval tuning 问题
2. 当前 Phase 2B 处于稳定节点，是插入新工作的好时机
3. 表格结构化完成后需要全量重建索引，但这已有成熟预案

## Approaches Considered

### Approach A: 最小可用版 — PDF 表格优先
- 只做 PDF 表格提取 + 表格感知 chunking
- 投入 S，1-2 天
- 直接打最大痛点，但其他格式暂时保持现状

### Approach B: 完整版 — 全格式表格结构化 (已选)
- PDF + DOCX + PPTX + XLSX 全部做表格结构化
- 投入 L，约 7 个工作日（PDF 2天 + DOCX/PPTX/XLSX 1天 + chunking 2天 + 集成测试 1天 + 重建+eval 1天）
- 彻底解决，一劳永逸
- 需要全量重建索引

### Approach C: 增量版 — 先恢复被删的表格文档
- 只针对电镀检验标准做手动处理
- 投入 M，2-3 天
- 风险最小但只解决一个文档

## Recommended Approach: 完整版 — 全格式表格结构化

### 实施步骤

#### Step 1: 引入表格提取依赖
- `pdfplumber` 用于 PDF 表格提取（比 camelot 更轻量，不依赖 Java）
- 升级现有 DOCX 表格提取逻辑，保留结构化数据
- PPTX 内嵌表格通过 XML 解析提取（`<a:tbl>` 元素）

#### Step 2: 统一表格中间格式
- 所有格式提取的表格统一转为 Markdown table 格式
- 同时存储结构化元数据到 chunk metadata：`{headers: [], rows: [], raw_markdown: str}`
- 结构化元数据本阶段仅存储，不参与检索评分。为后续 Phase 3 检索优化预留
- 在 chunk 的 metadata 中标记 `chunk_type: "table"`

#### Step 3: 表格感知 Chunking
- **新增 `TableAwareChunker`** 作为独立类，遵循现有 `SOPStructuredChunker.should_use()` 路由模式
- 表格作为一个整体 chunk，不拆分
- 如果表格超过 chunk_size，采用**保留表头 + 当前行**策略：每个拆分后的 chunk 包含完整表头和部分行，确保独立可读
- 表格 chunk 的 `retrieval_text` 包含 Markdown table 格式，保留行列关系
- CSV 文件作为表格整体处理，不单独做 chunk 拆分

#### Step 4: 文档解析器升级
- `document_parser.py` 的 PDF 解析路径增加表格检测
  - **表格 + 正文混排处理**：表格输出替换原文本流中的表格区域，保留周围正文顺序
  - **提取失败降级**：`pdfplumber` 提取失败时回退到当前纯文本行为，记录失败日志
  - **质量校验**：提取后检查行列数一致性（列数在行间一致、行数 > 0），不一致则降级
- DOCX 解析的 `_extract_docx_table()` 改为输出 Markdown table
- PPTX 解析增加 `<a:tbl>` 内嵌表格提取（当前 cell 文本已提取但丢失结构，改为按行列组织）
- XLSX 解析表头识别：第一行默认视为表头；如果第一行全部为数值，视为无表头

#### Step 5: Ingestion Service 集成
- 在 `ingestion_service.py` 中集成新的表格处理流程
- 保持与现有 `SOPStructuredChunker` 和 `TextChunker` 的兼容
- **关键验证**：重建前先测试表格提取后的 parsed text 不会干扰 `SOPStructuredChunker.should_use()` 的文档分类逻辑（Markdown 管道符 `|` 不应触发 clause pattern 匹配）

#### Step 6: 全量重建索引
- 按照 OPS_CHUNK_REBUILD_RUNBOOK.md 执行
- 重建后用 eval baseline 做 before/after 对比
- 重点关注表格类查询的召回率变化

#### Step 7a: Eval 样本恢复（本步骤范围）
- 恢复 disabled 的 4 条表格样本（fine-002, cross-007, lexical-002, cross-015）
- 验证电镀检验标准文档重新入库后这些样本能通过

#### Step 7b: Eval 样本扩充（后续 follow-up）
- 新增 5-10 条针对不同格式表格的查询样本
- 建立"表格查询 vs 非表格查询"的分组指标

### 文件变更范围

| 文件 | 变更类型 |
|------|----------|
| `backend/app/rag/parsers/document_parser.py` | 修改：增加 PDF 表格提取，升级 DOCX/PPTX 表格提取 |
| `backend/app/rag/chunkers/text_chunker.py` | 修改：增加 `TableAwareChunker` 类，遵循 `should_use()` 路由模式 |
| `backend/app/services/ingestion_service.py` | 修改：集成新的表格处理流程 |
| `requirements.txt` | 修改：新增 pdfplumber |
| `eval/retrieval_samples.yaml` | 修改：恢复 disabled 表格样本 |
| `backend/tests/test_document_ingestion.py` | 修改：增加表格处理测试 |
| `RETRIEVAL_OPTIMIZATION_BACKLOG.md` | 修改：记录表格结构化工作 |

### 与现有计划的交互

- 表格结构化作为 Phase 2B 的一个独立子工作流
- 完成后回到 retrieval optimization backlog 继续推进
- 不影响已有的 Phase 2A 成果和 eval 基线

### 降级与回滚

1. **单文档降级**：`pdfplumber` 提取失败时，自动回退到当前纯文本提取
2. **全量回滚**：如果重建后 eval baseline 回退，按 OPS_CHUNK_REBUILD_RUNBOOK.md 回滚索引
3. **路由冲突**：如果 Markdown table 管道符干扰 `SOPStructuredChunker.should_use()`，在 parsed text 送入 chunker 前先做标记替换

## Open Questions

1. 是否需要前端表格渲染组件，还是纯文本展示暂时够用？（非 blocker，可在实现后再决定）

## Success Criteria

1. 电镀检验标准文档能重新入库并通过 eval（恢复 fine-002, cross-007 等 4 条样本）
2. 表格类查询的 top1 accuracy >= 80%
3. 现有 verified-only eval baseline 不回退
4. 表格 chunk 保留完整的行列关系，可读性优于当前的 tab 分隔文本

## What I noticed about how you think

- 你选择了完整版而不是最小版。在确认了表格是真实痛点之后，你愿意投入更大精力一次到位，而不是分期修补。这反映了你对系统质量的重视。
- 你先要求做影响评估再决定方案。没有凭直觉冲进去实现，而是先量化问题的规模和影响。这是工程上正确的做法。
- 你的 retrieval optimization plan 写得非常细致和有纪律性（phases, gates, rollback plans, eval baselines）。这种严谨的方法论在整个行业都不常见。
